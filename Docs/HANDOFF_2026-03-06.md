## Root Cause Summary

- The original `Unknown or unsupported command type` failures were misleading: the Python catalog was fine, but Unity never registered the bridge handlers because `LiveV2V3Tools.cs` was failing to compile.
- The decisive diagnostic signal came from the live Unity console after refresh, not from static editor diagnostics.
- After compile recovery, the remaining failures were contract-shape and performance issues rather than missing registration.
- The last live blocker was narrower: `focus_hierarchy` still failed after the bridge recovery because its string-based Unity resolver path was weaker than `find_gameobjects`, so inactive/path-based scene objects could be discovered by smoke setup but not focused in the Hierarchy.
- For whole-editor screenshots, the most reliable path was not Unity-side window capture. The final working implementation intercepts local HTTP editor-capture requests in the Python server and uses an OS-level client-area capture of the Unity editor window on Windows.

## Known Operational Gotchas

- Duplicate GameObject names can route VFX actions to the wrong object when using `by_name`.
- For negative instance IDs, use `--` before the target ID in CLI calls.
- Square-card particle visuals are typically material/render-mode related; use textured alpha falloff and non-billboard settings where possible.
- Unity domain reloads during plugin recompilation can temporarily drop the editor instance from the local MCP server. Retry after reconnect before treating this as a persistent transport failure.
- For the bridged V2/V3 tool surface, prefer keeping Python/server payload naming aligned with Unity aliases. Snake_case compatibility is now handled for the playbook and benchmark entry points.
- Whole-editor capture is currently a local Windows HTTP capability. It depends on selecting a visible/restorable Unity editor window and should not be documented as reliable for minimized or remote-hosted sessions.
